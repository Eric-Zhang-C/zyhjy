gem5 page table, MMU, cache and memory

发表于 2023-08-24 更新于 2023-08-25 分类于 体系结构 ， gem5 阅读次数：
本文字数： 2.1k 阅读时长 ≈ 3 分钟

page table

SE模式

在SE模式下，gem5的页表实现为 class EmulationPageTable，实际承担页表项查找的数据结构是typedef std::unordered_map<Addr, Entry> PTable,相比传统基数页表的实现简化了许多。

添加新页表项的函数为EmulationPageTable::map，只会在两种情况被触发：

1. Process init

我们以Arm64架构为例。ArmProcess64初始化时，要将程序的.test, .data, .bss, heap, stack映射到虚拟地址空间(Guest Virtual Address)，相应的需要为这些虚拟地址分配物理地址(Guest Physical Address)，并将映射关系通过页表记录下来。

注： Guest Physical Address Space也是在Process初始化时mmap从host机器上获得，相关函数为PhysicalMemory::createBackingStore，详见本文第四部分memory。

具体调用流程：

ArmLinuxProcess64::initState -->> TranslatingPortProxy::tryOnBlob -->> Process::allocateMem -->> EmulationPageTable::map

2. page fault

另外一种新增页表项的原因则是由于程序执行时的stack与heap的增长，在IEW阶段触发了page fault；commit阶段会检查ROB中的指令，如果发现fault则调用o3::CPU::trap陷入内核处理fault。

具体调用流程：

Commit::tick -->> Commit::commit -->> Commit::commitInsts -->> Commit::commitHead -->> CPU::trap -->> GenericPageTableFault::invoke -->> Process::fixupFault -->> MemState::fixupFault -->> Process::allocateMem -->> EmulationPageTable::map

FS模式

FS模式下，页表由guest kernel管理，gem5不再参与页表初始化映射。当发生page fault时，GenericPageTableFault::invoke函数通过remote gdb由guest kernel处理异常。
