Lock-free Programming

Lock的问题

在谈无锁编程之前，我们首先来谈一谈锁存在哪些问题：

  • 死锁Deadlock：导致程序卡住。
  • 优先级倒置Priority inversion：低优先级进程抢占了高优先级进程需要的锁。
  • 排序Convoying：有多个进程需要以大致相同的顺序获得一系列锁，但是一个慢进程获得了第一个锁，其他进程不得不等待。
  • 异步信号安全Async-signal safety :信号处理函数不能使用lock-based primitives，因为信号处理函数允许同一线程重入，第二次进入时锁可能被第一次进入的堆栈持有，发生死锁。
  • Kill-tolerance：如果线程在持有锁的情况下被终止或崩溃，可能会出现死锁、数据不一致、资源泄漏。
  • 被抢占Pre-emption tolerance：如果线程在持有锁的情况下被抢占（pre-empted），可能会发生资源阻塞、死锁、锁竞争。
  • 性能performance：老生常谈的问题。

Lock-free Programming(无锁编程)

what is it

所谓无锁编程，是在不使用synchronization primitives（如mutex）的情况下实现线程安全接口。这常常需要硬件的支持。

人们常把无锁编程描述为在编程中不使用mutex/lock。这是正确的，但这只是其中的一部分。本质上，无锁是用来描述一些代码的属性，而不会详细说明该代码是如何编写的。

如果程序中的某个部分满足以下条件，则该部分可以被正当地视为无锁。相反地，如果代码中的某个部分不满足这些条件，则该部分不是无锁的。

从这个意义上讲，"无锁" 中的 "锁" 并不是直接mutex/lock，而是指在某种程度上"锁住" 整个应用程序的可能性。所以无锁编程，可以理解为wait-free。

无锁编程相关知识

事实证明，当您尝试满足无锁编程的非阻塞条件时，会产生一整套技术：原子操作、内存屏障、避免ABA问题等等。

那么，这些技术之间如何关联呢？为了说明这一点，请看以下流程图。我将逐一详细解释每个技术。

Atomic Read-Modify-Write Operations

原子操作是以一种看似不可分割的方式操纵内存的操作：没有线程可以观察到操作的一半完成。在现代处理器上，许多操作已经是原子的。例如，对齐的简单类型的读写通常是原子的。

Read-modify-write(RMW) Operations更进一步，允许您以原子方式执行更复杂的事务。当无锁算法必须支持多个写入者时，它们特别有用，因为当多个线程尝试在相同地址上进行RMW操作时，它们会有效地排成一排，并逐个执行这些操作。

RMW Operations的示例如C++11中的 std::atomic<int>::fetch_add。请注意，C++11原子标准并不保证在每个平台上都实现为无锁操作，因此最好了解您的平台和工具链的能力。您可以调用 std::atomic<>::is_lock_free 来确认。

RMW 操作在不同的 CPU 家族中是通过不同的方式来支持的。

  • x86/64 和 Itanium 架构通过 Compare-And-Swap (CAS) 方式来实现；
  • PowerPC、MIPS 和 ARM 架构通过 Load-Link/Store-Conditional (LL/SC) 方式来实现；

例如在 x86 架构下，通过 LOCK 指令前缀可以使许多指令操作（ADD, ADC, AND, BTC, BTR, BTS, CMPXCHG, CMPXCH8B, DEC, INC, NEG, NOT, OR, SBB, SUB, XOR, XADD, and XCHG）变成原子操作，其中 CMPXCHG 指令可用于实现 CAS 操作。

ABA problem

ABA问题是在并发编程中经常会遇到的一种情况。它涉及到在某个线程执行期间，一个共享数据从开始状态 A 转变为中间状态 B，然后再回到状态 A。在这个过程中，另一个线程可能会访问这个数据，但由于数据的状态看起来没变过，它可能无法察觉状态的变化过程，从而可能会出现意外的行为。

为了更好地理解，考虑一个简单的例子：假设有两个线程 A 和 B 同时访问一个共享整数变量。初始状态下，这个变量的值是 1。线程 A 首先读取这个值，然后线程 B 抢先执行，将变量的值从 1 修改为 2，然后又迅速将其修改回 1。最后，线程 A 继续执行，它会发现变量的值没有变化，尽管实际上它经历了从 1 到 2，再到 1 的变化过程。

这种情况可能导致并发程序出现问题，因为线程 A 可能会基于变量的值作出错误的决策，由于它没有意识到变量的状态实际上已经发生过改变。

Sequential Consistency

顺序一致性意味着所有线程都在内存操作发生的顺序上达成一致，并且该顺序与程序源代码中的操作顺序一致。在顺序一致性下，不可能出现像我在之前的帖子中演示的内存重新排序等问题。

实现顺序一致性的一种简单（但显然不切实际）方法是禁用编译器优化，强制所有线程在单个处理器上运行。处理器永远不会以无序的方式看到其自己的内存效果，即使线程在任意时间被抢占并调度。

一些编程语言甚至为在多处理器环境中运行的优化代码提供了顺序一致性。在C++11中，您可以将所有共享变量声明为带有默认内存顺序约束的C++11原子类型。为了实现这一点，编译器在幕后输出了额外的指令 - 通常是内存屏障和/或RMW操作。

C++volatile关键字

在C++中，volatile 关键字用于告诉编译器，该变量可能在程序执行期间被意外地改变，从而阻止编译器对该变量的某些优化。主要的特性包括：

  1. 不进行优化：使用 volatile 关键字修饰的变量，编译器不会对其进行优化，以确保每次访问都从内存中读取值，而不是依赖于已缓存的值。这对于与硬件相关的寄存器或设备状态等情况可能是有用的。

  2. 不进行编译器优化重排：volatile 也会告诉编译器不要对其附近的代码进行重排优化，以保持预期的执行顺序。

然而，C++中的 volatile 并不能提供原子性或多线程同步的保证。在C++中，要实现多线程之间的同步，通常需要使用原子类型（std::atomic）或互斥锁（std::mutex）等机制。

Memory Ordering

正如流程图所示，每当您在多核（或对称多处理器）环境中进行无锁编程，并且您的环境不保证顺序一致性时，您必须考虑如何防止内存重排序。

在当今的架构中，强制正确的内存顺序的工具通常可以分为三类，它们防止了编译器重新排序和处理器重新排序：

  1. 轻量级的同步或屏障指令，我会在以后的帖子中谈到；
  2. 完整的内存屏障指令，我之前已经演示过；
  3. 具有获取（acquire）或释放（release）语义的内存操作。

获取语义防止程序顺序中后续的操作进行内存重排序，而释放语义防止程序顺序中前面的操作进行内存重排序。这些语义特别适用于生产者/消费者关系的情况，其中一个线程发布一些信息，而另一个线程则读取它。我以后也会在帖子中更详细地讨论这个。

Different Processors Have Different Memory Models

不同的CPU系列在内存重排序方面有不同的行为习惯。每个CPU厂商都会记录这些规则，并且硬件会严格遵循这些规则。例如，PowerPC和ARM处理器可以改变内存存储相对于指令本身的顺序，但通常来说，来自英特尔和AMD的x86/64系列处理器不会这么做。我们说前者的处理器拥有更宽松的内存模型。

在考虑无锁编程的移植性时，我们常常要考虑不同内存模型带来的差异。有一种方法是将这些特定于平台的细节抽象出来，尤其是在C++11为我们提供了一种编写可移植的无锁代码的标准方式。

lock-free stack's ABA problem

  1. Thread 1执行pop()，查看head，发现为A (A是一个指针, 如0x7f8d70983400)

  2. Thread 1将A->next（假设为B），存储下来

  3. Thread 2此时执行pop()成功, head现在变为了 B，A所指的内存同时OS回收了

    • 如果Thread 1现在被唤醒，CAS将失败，需要重新插入新节点，这是正常的。

  4. 但是 Thread 2 这时构造了一个新的节点，且内存地址恰好是之前的A，并将新节点push()

  5. Thread 1被唤醒后，CAS比较发现head仍为A，于是将head置为B，结束push()

    • 这导致Thread2插入的新节点丢失，造成内存泄漏
solution

Keep a ‘update count’ (needs ‘doubleword CAS’), 即对每次CAS操作进行编号

std::memory_order

std::memory_order是 C++11 标准引入的一组枚举常量，用于指定原子操作的内存顺序。在多线程编程中，原子操作是一种保证线程安全性的机制，而内存顺序定义了在多线程环境下，不同线程对内存的读写操作的顺序和可见性。

在使用原子操作时，你可以通过指定合适的 std::memory_order 值来控制操作的内存顺序。以下是 std::memory_order 枚举常量的几个主要成员：

  1. memory_order_relaxed：最轻松的内存顺序，不会保证任何顺序关系，只保证原子操作的单个操作是原子的。通常用于不需要强制顺序的情况，提供最大的性能。

  2. memory_order_consume：用于读取操作，对数据依赖关系的读取具有适当的顺序性，但是并不会保证与其他写操作的顺序关系。

  3. memory_order_acquire：用于读取操作，确保在当前读取操作之前的所有写操作都对其他线程可见。它提供了适当的顺序性，确保不会出现读取脏数据。

  4. memory_order_release：用于写入操作，确保在当前写入操作之后的所有读写操作都对其他线程可见。它提供了适当的顺序性，确保写入的数据能够被其他线程看到。

  5. memory_order_acq_rel：既可以用于读取操作也可以用于写入操作，提供了一个综合的顺序性，确保前后的读写操作都有适当的顺序关系。

  6. memory_order_seq_cst：最严格的内存顺序，保证了全局顺序性，所有线程对内存操作都具有相同的全局顺序。虽然提供了最高的顺序性，但可能会带来性能开销。

提供线程安全的malloc()与free()，且无需使用lock。

思路

为每个core分配一大块空间作为heap，使用两个无锁链表分别管理已分配和未分配的块；使用两次cas避免ABA问题，从而实现无锁的分配与释放内存。

RCU

RCU（Read-Copy Update）是一种用于实现多线程环境下高效读取操作的同步机制，特别适用于操作系统内核和其他需要在读取操作频繁的场景中。RCU的主要目标是在不阻塞读取线程的情况下允许并发更新数据结构，从而提高读取操作的性能。

RCU的工作原理如下：

  1. 读操作不加锁：在RCU中，读取操作是非阻塞的，读取线程可以直接访问数据结构，而无需获得任何锁。

  2. 写操作的延迟：当需要修改数据结构时，RCU并不立即更新原始数据，而是创建一个新的数据副本，并将修改应用到副本上。这可以避免在写操作时阻塞读取线程。

  3. 发布和回收：在写操作的新数据副本创建完成后，RCU会在合适的时机发布这个副本，使得读取线程可以看到新的数据。而旧数据的回收则是延迟的，只有当所有正在使用旧数据的读取线程都完成后，才会安全地进行回收。

  4. 避免ABA问题：为了避免类似ABA问题（比如CAS操作）的情况，在RCU中每个数据副本都有一个版本号，以区分不同的副本。

  5. 内存屏障：在发布新的数据副本和回收旧数据时，通常需要使用内存屏障来确保数据的一致性。

RCU的优点在于，它适用于读操作远远多于写操作的场景，并且避免了常规锁所带来的竞争和阻塞。但是，实现和正确使用RCU需要一些技术挑战，例如需要处理内存屏障、版本管理、回收等问题。

RCU最早是由Linux内核中引入的，用于提高对共享数据结构的并发读取性能。随着时间的推移，RCU的思想也被应用于其他领域。在操作系统和其他需要高效读取的场景中，RCU可以是一个有力的工具来解决并发性能问题。

  • 不能根治争用（contention）

    • 仍然可能发生饥饿现象
