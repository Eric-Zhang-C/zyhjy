进程的虚拟内存空间布局

首先在64位机器上看下 Linux 系统中进程虚拟内存空间的真实分布情况。

接下来将对每个区域进行介绍。

保留区

当linux运行一个可执行文件的时候，需要将可执行文件的代码加载到相应进程的虚拟地址空间，并初始化相应的页表。

而默认的代码段起始地址为0x400000，这使得[0x0, 0x400000)的空间往往不会包含任何内容，相应的页表也不会初始化，一旦访问将会触发segmentation fault。

为什么代码段起始地址为0x400000呢？

这是可执行文件的编译过程的最后一步链接(linking)决定的；当链接时， linker( ld)将编译生成的目标文件以及库文件整合在一起，解析符号引用，解决地址重定位，从而生成最终的地址空间统一的可执行文件。

linker( ld)的默认起始地址为0x400000，但如果开启PIE(Position Independent Executables)，如gcc -o pie -fpie demo.c，起始地址将发生改变。

之所以ld选择0x400000作为默认起始地址，是由于以下几个原因:

  1. linux系统将零地址视为非法地址，为了阻止访问零地址或使用偏移读取.text或.data/.bss内存（如array[i]，其中array为NULL），有必要设置一个足够大的无法访问空间(如4MB)作为哨兵。
  2. 系统开启透明大页(THP)(x86-64 page size有4KB,2MB,1GB三种情况)时,为了利用大页，节约页表项 项，起始地址最好2MB对齐。
  3. 低起始地址可以获得将近2GB的静态数据，所以不应该浪费过多的空间作为nullptr缓冲区。对于规模不大的程序来说，text+data+bss地址空间整体小于2GB，就可以使用32bit的相对地址覆盖(如%rax替换为%eax)，提高程序运行效率。

代码段(text)

在进程执行前，操作系统将代码载入到起始地址开始的一段虚拟地址空间内，该段空间就是代码段。代码段是Read Only类型。

在代码段跟数据段的中间还有一段不可以读写的保护段，它的作用是防止程序在读写数据段的时候越界访问到代码段，这个保护段可以让越界访问行为直接崩溃，防止它继续往下运行。

数据段(data) 与 .BSS

这两段都是用于存储全局变量和静态变量的。

那些在代码中被我们指定了初始值的全局变量和静态变量在虚拟内存空间中的存储区域我们叫做数据段。

那些没有指定初始值的全局变量和静态变量在虚拟内存空间中的存储区域我们叫做 BSS 段。这些未初始化的全局变量被加载进内存之后会被初始化为 0 值。
