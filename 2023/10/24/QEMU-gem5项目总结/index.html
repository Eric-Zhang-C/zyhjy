QEMU-gem5项目总结

QEMU-gem5项目总结

记录一下我在PPI实验室的第一个完整项目。

1. 背景

1.1 QEMU

QEMU是一个强大的功能模拟器。对QEMU而言，被模拟的平台被称为Guest，又称target；运行QEMU程序的平台称为Host。

1.1.1 QEMU全系统模式

QEMU全系统模式模拟了Guest OS需要的硬件，使得Guest OS可以在其中运行。

其中，全系统模式使用软件虚拟化技术实现了Guest机器的硬件设备（如CPU、Memory、IO等），为Guest OS提供了对应的ISA，使得完整的Guest OS可以运行在Host OS上。

指令集架构（Instruction Set Architecture，ISA）包含了基本数据类型,指令集,寄存器,寻址模式,存储体系,中断,异常处理以及外部I/O。

从Guest的视角来看，Guest代码运行在真实的Guest机器上；从Host视角来看，QEMU只是Host机器上诸多进程中的一个。那么，QEMU是怎么在Host机器上模拟Guset代码的执行呢？

QEMU全系统模式模拟了一个完整的Guest CPU，这是一个软件模拟的CPU（virtual CPU， vCPU）。Guest CPU提供了与Guest平台一致的ISA，因此Guest指令可以看作是在vCPU上执行。

那么QEMU如何帮助Guest CPU执行Guest指令？

答案是DBT。我们知道Guest指令不能直接在Host机器上执行，因此QEMU通过DBT将Guest指令翻译成Host指令再执行。

可以将Guest CPU看作是一个状态机，模拟Guest CPU其实就是模拟它的状态和状态转移，QEMU将这些状态保存在了Host机器的内存中。为了模拟Guest指令对vCPU所造成影响，DBT尽量用等价的Host指令来表示Guest指令所引发的状态转移：1）在执行一个BB之前，QEMU将需要使用的Guest CPU状态从内存中取出；2）执行DBT翻译后的BB指令；3）在执行一个BB之后，QEMU将被修改的Guest CPU状态写回内存。这样，QEMU作为Host机器上的进程，模拟执行了Guest指令，并维护了Guest CPU的状态。

1.1.2 DBT(动态二进制翻译)

Guest指令不能直接执行，而是需要通过DBT翻译成Host指令再执行。

DBT以BB(Basic Block,基本块)为粒度，将一个BB内的所有Guest指令翻译成Host指令。所谓BB，是指一段按顺序执行的代码块，在这段代码中除了最后一条指令，不会有其他改变控制流的指令（如bl,ret,call），也就是只有一个入口和出口的代码块。

DBT的过程是这样的：

  • QEMU通过一个翻译缓存来存储已经转换过的BB
  • 当Guest BB需要执行时，QEMU会先检查缓存中是否有对应的翻译好的Host BB
  • 如果有，QEMU就直接执行缓存中的Host BB
  • 如果没有，QEMU就会将Guest BB转换为TCG IR，然后再转换为Host BB，并存入缓存中，然后执行

1.1.3 内存虚拟化

Guest OS运行在QEMU中，它所管理的Guest Physical Address Space实际上是由QEMU提供的；而QEMU本身是Host机器上的一个进程，所以QEMU给Guest OS提供的Guest Physical Address Space实际上是Host Virtual Address Space的映射（二者往往只差一个offset，因为Guest Physical Address Space从0开始，而分配给Guest OS的Host Virtual Address Space的起始地址由Host OS决定；相应的，GPA与对应的HVA之间也只差一个offset）。

这里一共涉及四种不同类型的地址，GVA(Guest Virtual Address), GPA(Guest Physical Address), HVA(Host Virtual Address), HPA(Host Physical Address)。

地址翻译的过程如下：

  • QEMU通过TLB缓存GVA -> HVA的映射
  • 当进行地址翻译时，首先查找TLB中是否有对应的条目
  • 如果有，则翻译完成，直接使用HVA；这一过程也称为quick path
  • 如果没有，则进行page walk，完成GVA->GPA->HVA的翻译，将GVA->HVA存入TLB，并在接下来使用HVA；这一过程也称为slow path
  • 当Host机器执行QEMU时，Host机器会将HVA翻译成HPA，就像其他在Host机器运行的进程一样

1.1.4 运行机制

QEMU全系统模式启动时需加载内核(kernel)与文件系统镜像(file sysyem image)。具体来说，QEMU向Host OS申请一块Virtual Address Space作为Guest OS的Physical Address Space，将kernel与image载入Physical Address Space，使用DBT解释执行kernel与file sysyem代码，完成Guest OS的启动。

Guest OS启动之后，可以在用户态执行Guest程序，具体方式仍为DBT解释执行。如果在用户态程序运行时发生中断、Trap、异常(如缺页异常、除零错误等)等，QEMU会切换到内核态，执行对应的内核指令处理相应事件，处理完成后将切回用户态继续运行程序。这里的用户程序和内核指令都是通过DBT解释执行的。

1.1.5 QEMU的缺点

时间不准确

QEMU并没有模拟Guest机器的时钟，而是通过获取Host机器的时钟计算时间，这导致Guest程序运行时间与在真实Guest机器上相比有较大差距。QEMU本质上来说是一个功能模拟器，只模拟了Guest程序的正确执行，并不保证Guest程序的时序过程。

受限的CPU模型

QEMU实际上模拟了顺序执行的CPU模型，不支持流水线、O3模型，自然也不会进行分支预测。

1.2 gem5

相比于QEMU，gem5模拟器更加全面，它基于事件驱动模型，同时模拟了功能模块和时序模块。在这里，我们同样将被模拟的平台被称为Guest；运行gem5程序的平台称为Host。

1.2.1 gem5全系统模式(FS)

与QEMU的全系统模式相同，gem5全系统模式同样模拟了Guest机器的硬件设备（如CPU、Memory、IO等），为Guest OS提供了对应的ISA，使得完整的Guest OS可以运行在gem5提供的虚拟硬件平台上。

gem5具有对多个ISA的模块化支持，目前支持Arm、x86、MIPS、Power、RISC-V、SPARC等。所有这些ISA都可以与gem5的任何CPU模型一起使用。gem5包括四种不同的CPU模型：Simple CPU模型、TimingSimple模型、In-Order模型和O3(out of order)模型。

gem5中有两种不同的缓存系统：“Classic”和“Ruby”。Classic的特点是能够简单快速地搭建起缓存系统，但其牺牲了一定的真实性与可配置性。Ruby为内存子系统实现了详细的仿真模型，它可以建模包含或独占的缓存层次结构、各类缓存替换策略、缓存一致性协议实现、互连网络、DMA和内存控制器、定序器等。

1.2.2 解释执行

gem5使用解释执行的方式模拟Guest指令的执行。

以O3CPU为例，Guest指令在Fetch阶段被读取出来，然后根据Guset ISA进行解码，将解码后的指令信息储存在指令信息类对象中；指令对应的事件将在指令信息类对象到达特定的阶段被触发(如访存指令将在IEW阶段执行相应的访存操作)。

1.2.3 内存虚拟化

内存虚拟化的逻辑与QEMU相同，地址翻译同样完成GVA->GPA->HVA的翻译；不同的是，TLB中存储的是GVA->GPA的映射，而GPA->HVA的翻译由memory系统实现。

1.2.4 运行机制

gem5初始化的时候，内核和文件系统都加载到gem5内存里了，所有指令都会走流水线处理；和QEMU处理内核代码的逻辑是一致的。

在全系统模式下，gem5处理各种fault以及syscall分为三步：1）首先修改系统状态位和pc；2）读取内核代码进入流水线中执行；3）恢复系统状态位和pc。

此外，remote gdb是辅助调试定位问题的，用户可以使用gdb检查整个内核的状态，也可以检查gem5内存中的数据是否符合预期。

1.2.5 gem5的缺点

运行时间长

模拟相同的测试用例，gem5耗时平均是QEMU的30倍。

时序模块和功能模块紧耦合

gem5采用事件驱动的模拟模型，其中功能模块和时序模块紧密耦合难以区分，使得开发人员难以扩展新的功能或添加新的时序事件。

2. 项目设计

2.X 通信接口设计

2.X.1 进程间通信设计

抽象场景：
  1. 两个进程A和B通信
  2. A是生产者，B是消费者
  3. A端存在并发，具体来说，A进程有若干个子线程，每个子线程都会进行生产
  4. B端只有一个线程作为消费者
  5. B并不是完全的消费者，也存在B为生产者A为消费者的情况
思考：

设计共享内存是否需要考虑进程A内部是否存在并发，即：A内部的多线程并发访问的正确性应该由A本身的设计保证，还是由通信接口保证。

如果由A本身的设计保证，则需要在进程A和通信接口直接增加一层proxy，用于消除多个线程之间的竞争、保证对通信接口的访问是安全的。在这种设计下，通信接口IPC本身可以设计为单写单独的模式。

如果由通信接口保证，可以将通信接口IPC设计为多写多（单）读的模式，这样进程间通信接口（至少是生产者侧接口）可以支持并发访问，自然能保证A端多线程访问的安全。

出于可扩展性和性能的考虑，我决定将通信接口IPC设计为多写多读的模式，这样不但可以提高通信接口的并行度进而提升性能，也使得接口更加通用。

设计与实现：
  • 使用共享内存作为IPC的实现载体，其中整个共享内存区域被抽象为一块环形缓冲区（RingBuffer）。生产者通过通信接口对缓冲区进行写入，消费者通过通信接口对缓冲区的内容进行读取。
  • 对于并发访问的控制与同步，仅使用无锁数据结构或轻量级自旋锁，以最大程度上提高性能。
  • 对传输的二进制消息进行序列化/反序列化，提升共享内存的实际利用率。【11.30更新，暂不实现序列化与反序列化，因为在同一台机器上存储数据时，不会遇到网络分发拆解问题，可以直接存储结构体】
  • 使用模版参数设置队列结构体的size和align size
  • 没有单播和多播的考量，或者说，纯粹是单播

2.X.2 信息类型设计

一共有两种进程间通信内存：

  1. 主状态区：该内存区只记录用户程序的状态（未开始，正在运行，终止）等，以及其他相关信息，且对于每个CPU来说是唯一的，因此使用共享内存直接交互即可，并不需要使用队列。
  2. 消息区：QEMU每执行一个block，会将该block的相关信息作为一个结构体传输给gem5，由于存在很多个这样的结构体，而共享内存的空间是有限的，我们需要使用一个环形队列来不断重复的利用共享内存。
