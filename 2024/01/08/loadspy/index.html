Redundant Loads: A Software Inefficiency Indicator

4. LOADSPY IMPLEMENTATION

LOADSPY使用Intel Pin [60]来拦截每个内存加载操作。该工具利用插桩技术获取指令中要访问的有效地址M、访问长度δ，并将这对数据提供给运行时分析程序。在本节的其余部分，我们将讨论LOADSPY如何分别识别时间和空间加载冗余。

A. Detecting Temporal Load Redundancy

检测时间负载冗余需要两个信息：目标位置上的当前值\(v_{new}\)以及相同位置上的上次加载值\(v_{old}\)。运行时分析程序在执行原始程序的加载指令之前运行，获取内存范围[M：M + δ]上的当前值\(v_{new}\)。LOADSPY使用影子内存S来维护相同位置上的上次加载值。S[M]保存程序在位置M上最近一次加载的值。LOADSPY利用基于页表的方案[6]高效管理其影子内存。在运行时，分析程序从S[M：M + δ)获取\(v_{old}\)，从[M：M + δ)获取\(v_{new}\)。如果\(v_{old}\) = \(v_{new}\)，LOADSPY记录一次冗余加载的实例。所有字节必须匹配才能将加载标记为冗余。直观地说，子读取大小(sub-ride-size)冗余对程序员来说是不可操作的。但需要注意的是，\(v_{old}\)可能是由多次较短的读取、一次较长的读取或更常见的相同大小的单次读取生成的。如果不是冗余的，LOADSPY会使用新加载的值更新影子内存。此外，如果\(v_{old}\) ≠ \(v_{new}\)，LOADSPY记录一次非冗余加载的实例。 LOADSPY允许近似计算，通过允许浮点（FP）操作生成的新值与先前的值近似匹配。如果两个值的差异在阈值范围内，LOADSPY认为它们近似相等，并记录一次冗余加载的实例。阈值是可调节的；我们在实验中使用了1%。因此，LOADSPY将加载冗余分解为精确和近似两种情况。 LOADSPY将每个冗余加载实例（和非冗余加载实例）归因于两个操作中涉及的两个方面\(<C_{old}, C_{new}>\)，其中\(C_{old}\)是M上先前加载操作的调用上下文，\(C_{new}\)是M上当前加载操作的调用上下文。

下面的方程式计算执行中的时间负载冗余的比例：

\(\mathcal{R}_{prog}^{precise}=\frac{\sum_{i}^{}\sum_{j}^{}Redundant\ non-FP\ bytes\ loaded\ in\ ⟨C_i,C_j⟩}{\sum_{i}^{}\sum_{j}^{}non-FP\ bytes\ loaded\ in\ ⟨C_i,C_j⟩}\)

\(\mathcal{R}_{prog}^{approx}=\frac{\sum_{i}^{}\sum_{j}^{}Redundant\ FP\ bytes\ loaded\ in\ ⟨C_i,C_j⟩}{\sum_{i}^{}\sum_{j}^{}FP\ bytes\ loaded\ in\ ⟨C_i,C_j⟩}\)

一对调用上下文之间的加载冗余由以下方程式给出：

\(\mathcal{R}_{<C_{old}, C_{new}>}^{precise}=\frac{Redundant\ non-FP\ bytes\ loaded\ in\ ⟨C_{old}, C_{new}⟩}{\sum_{i}^{}\sum_{j}^{}non-FP\ bytes\ loaded\ in\ ⟨C_i,C_j⟩}\)

\(\mathcal{R}_{<C_{old}, C_{new}>}^{approx}=\frac{Redundant\ FP\ bytes\ loaded\ in\ ⟨C_{old}, C_{new}⟩}{\sum_{i}^{}\sum_{j}^{}FP\ bytes\ loaded\ in\ ⟨C_i,C_j⟩}\)

这些指标有助于识别代码区域（调用上下文对）中观察到最高冗余量的地方。

获取指令的运行时调用上下文：将运行时统计信息归因于一个平坦的概要文件（只有指令指针）无法为开发人员提供完整的洞察力。例如，将冗余加载归因于常见的库函数（例如strcmp）并不能提供很多信息，因为在大型代码库中可以从多个位置调用strcmp；某些调用甚至对用户代码来说可能并不明显。详细的归因要求将概要文件关联到完整的调用上下文：main():line->A():line->...->strcmp():line。LOADSPY要求在每个加载操作上获取调用上下文，因为每个加载操作都可能是冗余的或非冗余的。LOADSPY使用CCTLib [61]，它将调用上下文作为调用上下文树（CCT）[62]高效地维护，包括通过longjump、尾调用和异常的复杂控制流程。调用上下文作为唯一的32位整数提供，并记录在影子内存中（除了上次加载的值）。

B. Detecting Spatial Load Redundancy

对于数组和聚合对象，LOADSPY检查是否从同一对象的任何元素连续加载的两个加载（近似地）加载相同的值。例如，如果从数组a的连续两个加载，比如a[i]和a[j]，加载相同的值，LOADSPY将其标记为空间加载冗余的实例，并将其归因于相同的数据对象。 为了方便检测空间加载冗余，LOADSPY在影子内存中维护了从地址范围到活动数据对象的映射。与每个数据对象\(\mathcal{O}\)关联的是两个额外的信息：作为对\(\mathcal{O}\)执行的前一个加载操作的结果加载的单例值\(v_{old}\)，以及与对\(\mathcal{O}\)执行的前一个加载操作相关联的调用上下文\(C_{old}\)。在每次内存加载时，LOADSPY使用加载操作的有效地址在映射中查找它所属的数据对象。如果当前加载的值与之前对同一对象记录的值匹配，LOADSPY记录一次空间加载冗余的实例。冗余首先按层次地归因于所涉及的数据对象，然后归因于冗余中涉及的两个调用上下文。 LOADSPY为空间冗余提供了类似的整体程序和每对冗余的度量标准。此外，LOADSPY使用以下方程计算每个数据对象的度量标准，其中\(\mathcal{O}\)是一个数据对象。

\(\mathcal{R}_{\mathcal{O}}^{precise}=\frac{Redundant\ non-FP\ bytes\ in\ object\ \mathcal{O}}{\sum_{i}^{}non-FP\ bytes\ in\ object\ i}\)

\(\mathcal{R}_{\mathcal{O}}^{precise}=\frac{Redundant\ FP\ bytes\ in\ object\ \mathcal{O}}{\sum_{i}^{}FP\ bytes\ in\ object\ i}\)

在运行时获取数据对象的地址：LOADSPY监视静态和动态数据对象，但在空间冗余检测中忽略堆栈对象。在加载模块中分配的数据位于.bss段中，属于静态对象。每个静态对象在符号表中有一个命名条目，该条目通过与加载模块开头的偏移量标识对象的内存范围。静态对象的生存周期始于将包含它们的加载模块（可执行文件或动态库）加载到内存中，终止于卸载加载模块。LOADSPY拦截加载和卸载加载模块的过程，以监视静态数据对象的生命周期，并建立从对象的地址范围到相应数据对象的映射。 动态对象通过malloc函数系列（malloc、calloc、realloc）和mmap[63]之一进行分配。动态对象的内存在free和munmap时回收。LOADSPY拦截这些函数以建立从对象的地址范围到相应数据对象的映射。在运行时查询一个地址会获得对应静态或动态对象的句柄。句柄是一个唯一标识符，表示静态对象的对象名称或动态对象的分配调用上下文。

C. Identifying the Redundancy Scope

当冗余发生在相同的调用上下文中，即\(C_{old}\) = \(C_{new}\)时，保证在冗余位置周围存在一个循环。然而，在嵌套循环的代码中，不清楚冗余是在内部循环的迭代之间发生还是在外部循环的迭代之间发生，或者是在两者之间的其他循环之间发生。因此，有必要指出包围冗余对的语法范围。

我们通过一个名为MASNUM-2.2[65]的真实应用程序来说明范围的需要，如列表9左侧所示。LOADSPY发现91％的内存加载是冗余的，而贡献最大的是第6行。很容易推断，在内部do循环（第5行）的一个迭代中加载的x(iii+1)在下一个迭代中作为x(iii)再次加载。一个明显的优化是在内部do循环的迭代之间进行标量替换，以保留x(iii+1)（列表9右侧）。然而，这种优化并不能消除许多冗余加载。实际上，第1行的外部do循环重复搜索一个项目xx，而内部do循环执行线性搜索。结果是，内部循环在外部循环的两次迭代之间重复加载相同的一组元素。因此，加载冗余不仅存在于内部循环的迭代之间，还存在于外部循环的迭代之间。外部循环中的加载冗余突出了一个算法级别的低效性-重复的线性搜索。有了这个知识，我们可以用二分搜索替换线性搜索。更多细节请参见第VII-B节。

为了帮助开发人员集中注意力在加载冗余发生的范围上，我们在LOADSPY中加入了冗余范围功能。我们用符号\(\mathcal{S}\)表示冗余范围。在列表9中，冗余范围是外部do循环。下面我们详细介绍如何计算冗余范围。

我们首先扩展调用上下文以纳入循环信息。因此，加载操作的调用上下文如下所示：\(main() → loop_1 → f() → ... → loop_n → load_{old}\)。此外，LOADSPY维护一个64位全局时间戳计数器\(\mathcal{T}\)，在通过每个循环头部和每个加载操作时递增。因此，调用上下文快照可能如下所示：\(C_{old} = main() → loop_1[\mathcal{T} = 1] → f() → ... → loop_n[\mathcal{T} = 9] → load_{old}\)。我们将调用上下文E扩展为一个元组，即\(E_{old} = ⟨指向旧上下文的指针,\mathcal{T}_{old}⟩ = ⟨C_{old},10⟩\)。

列表10显示了一个简化的示例，其中冗余发生在内部循环（范围为\(loop_2\)）。在这种情况下，考虑以下一对调用上下文快照： \(E_{old} = ⟨main() → loop_1[\mathcal{T} = 1] → loop_2[\mathcal{T} = 2] → load_{old},\mathcal{T}_{old} = 3⟩\)

\(E_{new} = ⟨main() → loop_1[\mathcal{T} = 1] → loop_2[\mathcal{T} = 4] → load_{new},\mathcal{T}_{new} = 5⟩\)

请注意，与\(loop_1\)关联的计数器保持不变，而与\(loop_2\)关联的计数器已更改。每个加载操作维护一个指向调用上下文的指针，而不是整个调用上下文快照。因此，在检测到冗余时（即执行\(load_{new}\)时），\(loop_2[\mathcal{T} = 2]\)将已经更新为\(loop_2[\mathcal{T} = 4]\)；遍历\(C_{old}\)将找到\(\mathcal{T}_{loop_2} = 4\)。观察到\(\mathcal{T}_{old}\) < \(\mathcal{T}_{loop_2}\) < \(\mathcal{T}_{new}\) 。这个不变量表明\(loop_2\)是冗余发生的范围。对于\(\mathcal{T}_{loop_1}\)，同样的不变量不成立。

现在，考虑列表11中的一个简化示例，其中冗余发生在外部循环（范围为\(loop_1\)）。在这种情况下，考虑以下一对调用上下文快照：

\(E_{old} = ⟨main() → loop_1[\mathcal{T} = 1] → loop_2[\mathcal{T} = 2] → load_{old},\mathcal{T}_{old} = 3⟩\)

\(E_{new} = ⟨main() → loop_1[\mathcal{T} = 8] → loop_2[\mathcal{T} = 9] → load_{new},\mathcal{T}_{new} = 10⟩\)

请注意，与\(loop_1\)和\(loop_2\)关联的计数器都发生了变化。因此，当执行\(load_{new}\)时，\(loop_1[\mathcal{T} = 1]\)和\(loop_2[\mathcal{T} = 2]\)将分别更新为\(loop_1[\mathcal{T} = 8]\) 和\(loop_2[\mathcal{T} = 9]\)；遍历\(C_{old}\)将找到\(\mathcal{T}_{loop_1} = 8\)和\(\mathcal{T}_{loop_2} = 9\)。观察到\(\mathcal{T}_{old}\) < \(\mathcal{T}_{loop_1}\) < \(\mathcal{T}_{loop_2}\) < \(\mathcal{T}_{new}\) 。

具有满足此不变量的最小\(\mathcal{T}\)值的循环，即\(loop_1\)，是冗余的作用范围。如果存在另一个包围循环，比如\(loop_0\)，它的计数器将不满足此不变量。

主张1. 对于冗余⟨⟨C，\(\mathcal{T}_{old}\) ⟩，⟨C，\(\mathcal{T}_{new}\) ⟩⟩，存在C中最外层的包围循环i，满足\(\mathcal{T}_{old}\) < \(\mathcal{T}_{loop_i}\) < \(\mathcal{T}_{new}\) 。 证明。首先，\(\mathcal{T}_{loop_i}\) 必须在(\(\mathcal{T}_{old}\) ，\(\mathcal{T}_{new}\) )的范围内，因为循环i是冗余的作用范围；否则，循环i将无法包围冗余的加载实例。接下来，假设存在C中的另一个循环j，使得\(\mathcal{T}_{old}\) < \(\mathcal{T}_{loop_j}\) < \(\mathcal{T}_{loop_i}\) < \(\mathcal{T}_{new}\) ，但循环j不是冗余的作用范围。循环i和j不能是对等的循环，因为它们都在同一个上下文C中。那么一个循环必须包围另一个循环。（1）如果循环i包围循环j，则循环j的计数器在循环i的计数器递增后至少递增了一次，这与假设\(\mathcal{T}_{loop_j}\) < \(\mathcal{T}_{loop_i}\) 矛盾。因此，循环j不能嵌套在循环i内部。（2）如果循环j包围循环i，则循环i不再是满足\(\mathcal{T}_{old}\) < \(\mathcal{T}_{loop_i}\) < \(\mathcal{T}_{new}\) 的最外层循环。因此，循环j不能包围循环i。由于循环i和循环j既不是对等的循环，也不能相互嵌套，假设无效。因此，主张1成立。

实现冗余作用范围：LOADSPY结合静态和动态分析来计算每个冗余对的冗余作用范围\(\mathcal{S}\)。首先，LOADSPY在二进制代码中的每个循环头部（以及过程）处进行插装，以生成带有增强循环信息的调用上下文。它通过对二进制代码执行区间分析[66]来识别指令是否为循环头部，并将该信息集成到过程调用路径中。我们将带有循环信息的调用上下文称为扩展上下文。作为每个循环头部的一部分运行的运行时分析例程会递增64位时间戳计数器\(\mathcal{T}\)。作为每个加载指令的一部分运行的分析例程也会递增计数器\(\mathcal{T}\)。此外，为原始程序的每个字节的阴影内存扩展以保存计数器\(\mathcal{T}\)（除了32位调用上下文句柄和8位旧值）。

在每个检测到的加载冗余中，其中\(C_{old}\) = \(C_{new}\)，LOADSPY从根（main）向叶节点（加载指令）搜索调用路径，寻找第一个满足主张1的循环节点。这样的循环是当前加载冗余实例的冗余作用范围S。每个冗余实例记录三元组⟨\(C_{old}\)，\(C_{new}\),\(\mathcal{S}\)⟩。如果\(C_{old}\) ≠ \(C_{new}\)，则LOADSPY首先找到最低公共祖先（LCA）函数或包围\(C_{old}\)和\(C_{new}\)的循环，然后从根（main）向LCA搜索它们的公共调用路径，根据主张1获取S。为每个冗余实例计算冗余作用范围会引入较大的运行时开销。我们仅对给定的调用上下文对计算冗余作用范围一定次数（在我们的实验中为一次），这对大多数程序来说已经足够好了。

D. Handling Threaded Programs

LOADSPY维护每个线程的数据结构：调用上下文树、冗余配置文件等，因此对于多线程程序不需要并发控制。运行时对象映射作为一个无锁映射进行维护，允许并发查找。LOADSPY仅检测线程内的冗余，并忽略线程间的冗余（如果有的话）。

E. Reducing Profiling Overhead

LOADSPY可能会引入相对较高的运行时开销，大约是40-150倍。LOADSPY采用了一种突发抽样机制来控制其开销[67]。突发抽样包括连续监视一定数量的指令（WINDOW_ENABLE），然后在一定（更大）数量的指令后停止监视（WINDOW_DISABLE），并随时间重复这个过程。这两个阈值是可调的。根据我们的实验，使用1%的抽样率，WINDOW_ENABLE设置为100万，WINDOW_DISABLE设置为9900万，可以在开销和准确性之间取得良好的平衡。

F. Discussions

值得注意的是，由于硬件中的流水线、缓存和预取机制，冗余比例与潜在的性能收益之间并没有一对一的关系。LOADSPY不区分可操作的和不可操作的冗余，这是我们未来工作的一个课题。
